## 一、PRD 需求摘要

### 1.1 核心变化

```
个人版: 用户 ──创建──→ API 密钥 ──选择──→ 分组（模型）──→ 消费
企业版: 企业 ──管人──→ 成员 ──分配──→ API 密钥 ──选择──→ 分组（模型）──→ 消费
                                                                       ↑
                                                              企业资金池（独立）
```

**一句话**: 新增企业账号类型 + 企业独立资金池 + 成员管理 + API 密钥分配人员 + 企业维度财务聚合。

### 1.2 不改动的部分（PRD 明确）

| 模块 | 说明 |
|------|------|
| 用户注册/登录 | 完全不动 |
| 个人余额充值 | 完全不动，企业管理员个人充值复用同一套 |
| 个人套餐购买 | 完全不动，企业管理员个人套餐复用同一套 |
| LiteLLM 网关 | 需适配分组 M:N 路由（见 C1.6） |
| 支付系统 | 完全不动，企业充值复用同一套 |
| 消费优先级 | 完全不动（套餐→余额→报错），企业池同样优先级 |
| 运营方分组创建 | 完全不动 |

### 1.3 PRD 功能模块一览

| 模块 | 功能点 | 变更类型 |
|------|--------|:--:|
| **A1-A3** 企业账号管理 | 创建/编辑/启停/删除企业 | 🆕 新增 |

### 2.2 现有角色体系

```go
// backend/internal/domain/constants.go
const (
    RoleAdmin = "admin"   // 平台管理员
    RoleUser  = "user"    // 普通用户
)
```

**现状**: 仅 `admin` / `user` 两种平台角色。**无企业相关角色**。

> 角色体系扩展详见 [02-核心设计决策.md](./02-核心设计决策.md#决策-2用户角色体系)：
> - `users.role` 不变（仍只存 `"admin"` / `"user"`）—— 平台层身份
> - `enterprise_members.role` 新增 `"enterprise_admin"` / `"enterprise_member"` —— 企业层权限
> - `constants.go` 新增 2 个企业角色常量
> - 现有 5 处硬编码 `"admin"` 字符串需先统一为常量再扩展

## 四、风险评估

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| **分组 1:1→M:N** 是破坏性变更 | 所有查询 group_id 的代码都要适配 | 建 api_key_groups 中间表，保留旧字段兼容 |
| **计费分流**逻辑错误会导致错扣 | 用户/企业资金混乱 | 充分测试，pool_type 标记审计，增加审计日志 |
| **企业资金池独立**涉及新充值流程 | 充值入口混淆（个人vs企业） | 前端明确区分「个人充值」「企业充值」入口 |
| 企业成员解绑后 Key 禁用 | 在途请求可能被意外拒绝 | 短时缓冲后再禁用 |
| 企业停用/删除影响面大 | 全部成员 Key 即时失效 | 二次确认 + 通知 |
| 企业删除后余额处理 | 资金纠纷风险 | 本期余额不退，由运营方人工处理 |
