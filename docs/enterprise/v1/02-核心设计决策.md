## 设计原则

> **核心原则**：一个用户最多属于一个企业。企业拥有独立的余额和套餐池（企业资金池），与管理员及成员的个人资金完全隔离。降低设计复杂度，但不阻断未来 M:N 扩展路径。

**三条资金线完全隔离**（PRD D3）：

```
管理员"李总"的资金构成：
  ├── 个人余额: ¥100（自己充值，个人用）
  ├── 个人套餐: GPT-4 月度包
  │
  ├── 企业资金池（XX科技）：
  │     ├── 企业余额: ¥5000（李总为企业充值）
  │     └── 企业套餐: Claude 企业版
  │
  ├── 自用 Key-A（assigned_to = NULL）
  │     → 扣李总个人余额/套餐
  │
  └── 分配给张三的 Key-B（assigned_to = 张三）
        → 扣 XX科技 企业资金池

企业成员"张三"的资金构成：
  ├── 个人余额: ¥50（自己充值）
  ├── 个人套餐: 无
  │
  ├── 自己创建的 Key-C（assigned_to = NULL）
  │     → 扣张三个人余额/套餐
  │
  └── 企业分配的 Key-B（assigned_to = 张三）
        → 扣 XX科技 企业资金池
```

## 决策 1：用户-企业关系 —— 1:1 约束

### 决策：**本期 `enterprise_members` 表约束一人一企业**，保留表结构以备未来 M:N

```sql
-- 唯一约束：一个用户只能有一条活跃的企业成员记录
CREATE UNIQUE INDEX idx_enterprise_members_user_active
    ON enterprise_members(user_id)
    WHERE status = 'active' AND deleted_at IS NULL;
```

### 理由

1. 避免"一个人同时是 A 企业管理员 + B 企业成员"的复杂性
2. 简化权限判断：用户要么是个人（无 enterprise），要么是某企业成员
3. 计费逻辑清晰：不存在"同一个 Key 到底走哪个企业资源池"的歧义
4. `enterprise_members` 表保留 `(enterprise_id, user_id)` 结构，未来只需移除唯一索引即可升级为 M:N

## 决策 2：用户角色体系

### 设计：两层角色，两种含义

角色分两层，`role` 都叫"角色"但在不同表中含义不同：

| 层 | 字段 | 含义 | 值域 |
|---|------|------|------|
| **平台层** | `users.role` | 用户在整个平台中的身份 | `"admin"` / `"user"` |
| **企业层** | `enterprise_members.role` | 用户在某企业内的权限角色 | `"enterprise_admin"` / `"enterprise_member"` |

> ⚠️ `users.role` 不加新值。企业身份通过 `enterprise_members` 表体现，不写入 `users` 表。

### 为什么是两个字段而不是合并？

平台身份和企业身份是**正交的**两个维度：

- **平台管理员**（`users.role = "admin"`）：管理全部企业，其权限来自平台层，与 `enterprise_members` 无关
- **企业管理员**（`enterprise_members.role = "enterprise_admin"`，但 `users.role = "user"`）：在平台上仍是普通用户，只在自家企业内有管理权

如果合并成一个字段：`users.role = "enterprise_admin"`，就无法表达"这个人到底是平台管理员还是企业管理员"——它们是互斥的。分两层就不会互斥，未来如果需要也可以组合。

### 常量定义

```go
// backend/internal/domain/constants.go
const (
    // 平台层角色（users.role，不变）
    RoleAdmin = "admin"   // 平台管理员
    RoleUser  = "user"    // 普通用户

    // 企业层角色（enterprise_members.role，新增）
    EnterpriseRoleAdmin  = "enterprise_admin"   // 企业管理员
    EnterpriseRoleMember = "enterprise_member"  // 企业普通成员
)
```

全部定义在同一文件、同一 `const` 块，统一管理。

### enterprise_members.role 值说明

| 值 | 常量 | 含义 | 能做什么 |
|----|------|------|----------|
| `"enterprise_admin"` | `EnterpriseRoleAdmin` | 企业管理员 | 管成员/管 Key/企业充值/看企业财务 |
| `"enterprise_member"` | `EnterpriseRoleMember` | 企业普通成员 | 接收企业分配的 Key，看个人用量 |

### 用户四类状态

| # | users.role | enterprise_members.role | 含义 |
|:--:|:--:|:--:|------|
| ① | `admin` | —（不存在） | 平台管理员，管理所有企业（仅限操作 parent_id=0 的顶级企业） |
| ② | `user` | —（不存在） | 个人用户，无企业归属 |
| ③ | `user` | `enterprise_admin` | 企业管理员，管理自身企业 |
| ④ | `user` | `enterprise_member` | 企业成员，接受企业分配的 Key |

### 鉴权方式

```
                     用户请求
                        │
                  users.role == "admin"?
                   /              \
                 是                否
                 │                 │
           平台管理员权限      查 enterprise_members
            (管理全部企业)          │
                      enterprise_members.role?
                      /                    \
                "enterprise_admin"    "enterprise_member"
                      │                    │
                企业管理员权限          企业成员权限
              (管理自身企业)          (查看个人+分配Key)
```

### 未来扩展（如财务角色）

`enterprise_members.role` 是 `varchar(20)`，新增角色不需要改表结构：

```go
// 未来只需加一行常量
EnterpriseRoleFinance = "enterprise_finance"  // 企业财务（只看财务数据）
```

再新增对应的鉴权中间件即可，架构零变化。

## 决策 3：企业资金池 —— 独立于管理员个人余额

### 决策：**企业拥有独立的余额和套餐池**，与管理员的个人资金完全分离

### 企业资金池设计

```
enterprises 表：
  balance          DECIMAL(20,8)   # 企业余额（独立于 users.balance）
  total_recharged  DECIMAL(20,8)   # 企业累计充值

enterprise_subscriptions 表：       # 企业套餐（独立于 user_subscriptions）
  enterprise_id    BIGINT          # 企业 ID
  group_id         BIGINT          # 分组
  plan_id          BIGINT          # 套餐计划
  starts_at / expires_at           # 有效期
  daily/weekly/monthly_usage_usd   # 已用量
  status                           # active/expired/suspended
```

### 与旧方案的对比

| 维度 | V1.1（已废弃） | V1.2（PRD 对齐） |
|------|:--:|------|
| 企业资金池 | 管理员个人余额 | **enterprises.balance + enterprise_subscriptions** |
| 企业充值对象 | 管理员 users 账户 | **enterprises 记录** |
| 企业消费扣款 | 扣管理员余额 | **扣企业 balance / enterprise_subscriptions** |
| 管理员自用Key | 扣"企业资金池"（混乱） | **扣管理员个人余额** |
| 企业财务视图 | 查管理员余额 | **查 enterprises.balance + 企业套餐** |

### 充值流程

- 个人用户充值：`用户 → 支付 → users.balance += 金额`（**不变**）
- 企业管理员充值：`管理员 → 选择"为企业充值" → 支付 → enterprises.balance += 金额`
- 企业管理员买套餐：`管理员 → 选择"为企业购买" → 支付 → 创建 enterprise_subscriptions`
- 管理员个人充值：`管理员 → 选择"个人充值" → 支付 → users.balance += 金额`（与普通用户一致）

> **充值入口**：前端在财务页提供「个人充值」「企业充值」两个入口，后端同一套支付流程，仅记账目标不同。

## 决策 5：计费归属核心逻辑

### PRD 对齐的三条线分流

```go
// 每次 API 调用时判断
func ResolveBilling(key *APIKey) (*BillingContext, error) {
    if key.AssignedTo == nil {
        // 个人 Key / 管理员自用 Key → 扣创建者个人余额/套餐
        return &BillingContext{
            OwnerUserID:        key.UserID,
            EnterpriseID:       nil,
            PayerUserID:        key.UserID,           // 扣创建者
            SubscriptionUserID: key.UserID,           // 扣创建者套餐
            PoolType:           "personal",
        }, nil
    }

    // 企业分配的 Key → 扣企业资金池
    member := findEnterpriseMember(key.AssignedTo)
    enterprise := findEnterprise(member.EnterpriseID)

    if enterprise.Status != "active" {
        return nil, errors.New("企业已停用")
    }

    return &BillingContext{
        OwnerUserID:          key.UserID,            // Key 所有者（管理员）
        EnterpriseID:         enterprise.ID,         // 消费归属企业
        PayerUserID:          0,                     // 不从个人扣
        SubscriptionUserID:   0,                     // 不从个人套餐扣
        EnterprisePayerID:    enterprise.ID,         // 扣企业余额
        ConsumerUserID:       member.UserID,         // 实际使用者
        PoolType:             "enterprise",
    }, nil
}
```

### 扣费流程

```go
func Consume(ctx *BillingContext, cost float64) error {
    if ctx.PoolType == "enterprise" {
        // 1. 尝试从企业套餐扣费（查 enterprise_subscriptions）
        // 2. 企业套餐不足 → 从企业余额扣费（enterprises.balance）
        // 3. 都不足 → 返回错误
    } else {
        // 个人池（不变）：套餐 → 余额 → 报错
    }
}
```

### 消费写入 usage_logs

```
usage_logs 每条记录：
  user_id        → 实际调用 API 的用户（成员或管理员）
  enterprise_id  → 有值 = 企业消费, NULL = 个人消费
  api_key_id     → 使用的 Key
  total_cost     → 金额
  pool_type      → 🆕 "personal" | "enterprise"（便于审计追溯）
```

### Key 消费归属汇总

| Key 类型 | created_by | assigned_to | 扣谁的钱 | 资金池 | 记谁头上 |
|----------|:--:|:--:|------|:--:|------|
| 个人 Key | 任意用户 | NULL | 用户自己 | 个人池 | 个人消费 |
| 管理员自用 Key | 管理员 | NULL | 管理员自己 | 管理员个人池 | 管理员个人消费 |
| 企业分配 Key | 管理员 | enterprise_members.id | **企业** | 企业资金池 | 企业消费(enterprise_id) |

## 决策 7：企业控制台复用

### 决策：**完全复用现有前端控制台**，基于用户是否属于企业动态调整视角（不变）

```
用户登录 → 进入同一套前端：
  │
  ├── 无企业归属（enterprise_members 不存在）
  │     → 现有个人版行为，完全不变
  │
  ├── 企业管理员（enterprise_members.role = enterprise_admin）
  │     → 看到个人面板 + 🆕 企业管理入口
  │     → API Key 页：全企业 Key 列表
  │     → 财务页：个人财务 + 🆕 企业财务（独立显示企业余额/套餐）
  │     → 充值页：个人充值 + 🆕 企业充值入口
  │
  └── 企业普通成员（enterprise_members.role = enterprise_member）
        → 看到个人面板 + 企业标识
        → API Key 页：个人 Key + 企业分配的 Key
        → 财务页：个人消费 + 企业名称标注
```

## 决策 8：数据库迁移策略

### api_keys.group_id 1:1→M:N

```sql
-- 1. 创建中间表
CREATE TABLE api_key_groups (
    id          BIGSERIAL PRIMARY KEY,
    api_key_id  BIGINT NOT NULL REFERENCES api_keys(id),
    group_id    BIGINT NOT NULL REFERENCES groups(id),
    created_at  TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    UNIQUE(api_key_id, group_id)
);

-- 2. 数据迁移
INSERT INTO api_key_groups (api_key_id, group_id)
    SELECT id, group_id FROM api_keys WHERE group_id IS NOT NULL;

-- 3. api_keys.group_id 保留但标记 deprecated
```

### 其他表变更

```sql
-- enterprises 增加企业资金池字段 🆕
ALTER TABLE enterprises ADD COLUMN balance DECIMAL(20,8) DEFAULT 0;
ALTER TABLE enterprises ADD COLUMN total_recharged DECIMAL(20,8) DEFAULT 0;

-- 🆕 企业套餐表
CREATE TABLE enterprise_subscriptions (
    id              BIGSERIAL PRIMARY KEY,
    enterprise_id   BIGINT NOT NULL REFERENCES enterprises(id),
    group_id        BIGINT NOT NULL REFERENCES groups(id),
    plan_id         BIGINT NOT NULL REFERENCES subscription_plans(id),
    starts_at       TIMESTAMPTZ NOT NULL,
    expires_at      TIMESTAMPTZ,
    status          VARCHAR(20) NOT NULL DEFAULT 'active',
    daily_usage_usd    DECIMAL(20,10) DEFAULT 0,
    weekly_usage_usd   DECIMAL(20,10) DEFAULT 0,
    monthly_usage_usd  DECIMAL(20,10) DEFAULT 0,
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at      TIMESTAMPTZ
);

-- usage_logs 增加企业归属
ALTER TABLE usage_logs ADD COLUMN enterprise_id BIGINT;
ALTER TABLE usage_logs ADD COLUMN pool_type VARCHAR(20) DEFAULT 'personal';  -- 🆕
CREATE INDEX ON usage_logs(enterprise_id, created_at);

-- api_keys 增加企业字段
ALTER TABLE api_keys ADD COLUMN assigned_to BIGINT;       -- → enterprise_members.id
ALTER TABLE api_keys ADD COLUMN usage_purpose VARCHAR(200);
ALTER TABLE api_keys ADD COLUMN bound_tool VARCHAR(50);
```
